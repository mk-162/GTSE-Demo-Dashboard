## 2. M1 — Server data facade (commit `f1ffc96`)

The session's anchor commit. Master plan §6.

**Goal:** zero `lib/mock-data` imports outside `lib/mock-data/*` itself. Every page, route, and AI surface reads through `lib/data/` (the typed facade).

**What landed:**
- `lib/data/contracts.ts` — `DataLayer` interface with 15 methods + re-exported types. `import "server-only"` so a Client Component accidentally importing it fails the build instead of bundling server code into the browser.
- `lib/data/index.ts` — `getData()` selects impl by runtime (`process.env.NEXT_RUNTIME === "edge"`) + `DATA_SOURCE` env var, via **dynamic** `import()`. Master plan §4.7 — static imports of both impls would defeat the bundle split.
- `lib/data/impl/memory.ts` — wraps existing mock data behind the contract. Every method is async (normalises shape so swapping to Postgres later is invisible).
- `lib/data/impl/postgres-{edge,node}.ts` — stubs throwing `not implemented` for now (filled in by `433d79b`).
- `lib/data/contracts.test.ts` — 18 contract tests, all green against `memoryImpl`.
- `vitest` + `swr` installed; `vitest.config.ts` aliases `server-only` to an empty shim under test.

**Page refactor pattern:**
- All 11 pages converted to **Server Component shell + Client Component view**.
- Server reads `searchParams.region` (URL override) or `whale_region` cookie, calls `getData()`, fetches data, passes as props.
- Client view does interactivity (sort, filter UI, etc.).
- Targets page is the special case: server provides initial filtered list + ranges + owners; client uses SWR to refetch via the new `/api/internal/companies` on slider changes (debounced).
- Account detail: server fetches `companyById` + `ordersForCompany`, passes to client view.

**Region migration: localStorage → cookie.**
- `lib/region.ts` (server) — `resolveRegion(searchParams)` reads URL param, then cookie, then defaults to UK.
- Layout reads cookie via `cookies()` and passes `initialRegion` to `RegionProvider`.
- Provider's `setRegion` writes the cookie + calls `router.refresh()` so server components re-render with the new region.
- One-time migration effect lifts a legacy localStorage value into the cookie on first load (returning users don't lose their selection).

**API surfaces:**
- All 9 `/api/v1/*` routes refactored to use `getData()`. Still Edge runtime — M5 will move them to Node.
- `/api/chat` and `/api/insights/regenerate` await `getData()` via the now-async `ai-context` helpers.
- New `/api/internal/companies` and `/api/internal/companies/name-index` — Node runtime, password-cookie-authed via middleware. Used by Targets SWR refetch and the markdown linkifier respectively.

**AI surfaces:**
- `lib/ai-context.ts` — `buildDataContext` + `buildInsightPrompt` are now async and route through the facade.
- `MessageMarkdown` accepts an optional `nameToIdMap` prop; bold spans whose lower-cased text matches a key auto-linkify to `/account/[id]`. Fed by server-rendered map on pages and SWR fetch in the chat panel.
- `InsightBanner` forwards `nameToIdMap` to `MessageMarkdown`.

- `lib/db/migrate.ts` — idempotent runner. Bootstraps `app` schema + `app.migrations` tracking table inline before applying file-based migrations. Safe to call on every cron invocation.

**Schema migrations 001–004:**
- `001_schemas.sql` — `pgcrypto` extension + 5 schemas (`raw_hubspot`, `raw_netsuite`, `staging`, `marts`, `app`).
- `002_raw_hubspot.sql` — 5 object tables (companies, deals, line_items, contacts, engagements) + 2 association tables (deal↔company, deal↔line_item). Append-only, JSONB payload, `(id, modified)` primary key + `(modified DESC)` index.
- `003_app_tables.sql` — `ingestion_runs`, `ingestion_cursors`, `dashboard_insights`, `api_access_log`.
- `004_raw_netsuite.sql` — 5 record-versioned tables (customers, items, sales_orders, purchase_orders, vendors) + 1 point-in-time table (`inventory_snapshots`).

**Ingestion modules:**
- `lib/ingest/hubspot-client.ts` — wraps `@hubspot/api-client` with token check + retry config.
- `lib/ingest/upsert.ts` — bulk INSERT ... ON CONFLICT DO NOTHING into `raw_hubspot.<table>`. Allowlisted table name (`HUBSPOT_TABLES` set) to avoid templating arbitrary identifiers. JSONB payload `JSON.stringify`'d explicitly because postgres-js doesn't auto-serialize objects.
- `lib/ingest/cursor.ts` — read/write `app.ingestion_cursors` per (source, object_type). Cursor only advances on success.
- 5 pull modules (`pull-companies`, `pull-deals`, `pull-line-items`, `pull-contacts`, `pull-engagements`) — paginated search by `hs_lastmodifieddate`; max-modified tracked in-loop and committed to cursor on success.

**Cron route:**

Runs migrations first, inserts `app.ingestion_runs` row with `status='running'`, calls all 6 pulls in dependency order, updates row with `status='success'` + counts on success or `status='failed'` + error JSON on exception.

## 7. Three judgment calls flagged for review tomorrow

These are areas where I made a defensible-but-reviewable decision today. None are blockers but each is worth a glance before activation:

### 7.1 `marts.dim_customer` engagement-derived columns are NULL

Affected columns: `last_engagement_date`, `email_opens_l60d`, `active_contacts`, `days_since_last_engagement`, `buyer_intent_active`.

**Why NULL:** these need a `staging.engagement` view that joins `raw_hubspot.engagements` + `raw_hubspot.contacts` to `staging.customer`. Phase 0 §A8 confirms the Sensitive Data flag is off and §A7 confirms engagement volume; until those are answered I deferred writing the staging view.

**Impact today:** dashboard tolerates NULL — account detail page renders "—" instead of 0/0/0. The health score formula uses these inputs but with NULLs falls back to lapse-only (still directional, just less full-fidelity).

**Real fix tomorrow** (after Phase 0 confirms property names): write `staging.engagement` view aggregating per company:
```sql
CREATE OR REPLACE VIEW staging.engagement AS
SELECT
  c.id AS customer_id,
  max(...) AS last_engagement_date,
  ... etc
FROM staging.customer c
LEFT JOIN raw_hubspot.engagements e ON ...
LEFT JOIN raw_hubspot.contacts ct ON ...
GROUP BY c.id;
```

Then update `dim_customer` to LEFT JOIN engagement and pull the columns through.

**Estimated effort:** ~30 min once Phase 0 confirms property names.

### 7.2 `top_3_reorder_skus` and `top_3_cross_sell_skus` are empty arrays

Affected: `marts.dim_customer.top_3_reorder_skus`, `top_3_cross_sell_skus`.

**Why empty:** computing these properly requires:
- `top_3_reorder_skus`: per-customer aggregation of `staging.fact_order_lines.sku_code` ordered by frequency * recency, top 3.
- `top_3_cross_sell_skus`: peer-basket lift analysis — for each customer, find top SKUs bought by industry peers but NOT by this customer.

The peer-basket query is a multi-CTE join with industry partitioning. Non-trivial but mechanical.

**Impact today:** account detail's "Whitespace SKUs" card shows "No cross-sell candidates" instead of 3 SKU codes. Reorder reminder cron (when written) wouldn't have a recommendation per-account.

**Real fix tomorrow:** write `marts.customer_top_skus` and `marts.customer_cross_sell_candidates` materialised views + LEFT JOIN them to `dim_customer`. ~45 min.

### 7.3 Engagements are NOT pruned by retention

`app.fn_retention_cleanup()` prunes ingestion_runs (90d), api_access_log (90d), inventory_snapshots (180d). Engagements stay indefinitely.

**Why:** PII (email body, attachment IDs) is stripped at ingestion (master plan §10.3 item 5; see `lib/ingest/pull-engagements.ts:PII_FIELDS_TO_STRIP`). Headers + timestamps remain — useful for long-tail signal (cohort analysis, "last touched 18 months ago" cohort).

**Impact:** if GDPR comes asking for per-record deletion, we need a `app.fn_delete_company_engagements(customer_id)` function. Not blocking but worth noting in the DPIA.
